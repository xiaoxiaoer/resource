资源置换业务评估测算

AI 驱动的自动化审核解题报告
2026 年 5 月
Part 1

业务背景

资源置换是停车场的核心商业合作模式——合作方以设备安装、停车券采买、车位置换等方式获取停车场运营权益。每一笔合作都需要运营人员对合作方提交的测算数据进行逐项审核，确保数据真实、风险可控。

停车券业务

合作方以约定折扣采买停车券，运营需核对后台临停、月票收入是否与测算表一致，验证消耗比例是否达标。

车位置换业务

合作方以设备安装置换车位经营权，运营需复核外部月卡定价、分成比例、合同期限等关键条款。

非艾科车场

无法直连后台数据，需依赖业务提供的信息复核，运营需自行搜集周边车场收费信息，结合位置、业态综合评估。

Part 2

五大核心挑战

现有审核流程高度依赖人工，效率低、易出错、标准不统一。

挑战 1

后台数据获取分散，缺乏统一报表

新车场周边数据需逐个登录 BEM（智泊云）后台获取临停收入、月票收入、票种信息，每个车场要手动切换、导出，耗时且容易遗漏。

挑战 2

各方数据格式不统一，分析困难

业务提交的测算表模板版本多样（停车券/车位置换/折扣采买），字段位置、sheet 结构差异大，人工对齐费时费力。

挑战 3

业务提交前缺乏自检，反复退回

业务人员提交的测算数据常存在字段缺失、数据异常等问题，运营复审时才发现退回，导致沟通反复、流程拉长。

挑战 4

承包期/产权方经营风险难以量化

需评估合作方承包期限是否覆盖合同期、企业社保人数是否异常、注册资本与经营规模是否匹配等，依赖运营经验判断。

挑战 5

缺乏智能初评体系

审核标准散落在运营人员经验中，无标准化流程；审核质量因人而异，新人上手周期长。

Part 3

解决方案总览

构建 "AI + 固化脚本" 的混合架构，让 LLM 像资深运营一样自主完成初审流程。

上传测算表
→
智能解析
→
数据完整性检查
→
BEM 数据抓取
→
企查查核查
→
AI 初审报告

核心思路：AI 编排 + Skills 分层

  • LLM（DeepSeek）作为 "AI 编排器"，通过 Function Calling 自主决定审核节奏
  • 数据获取层用 Playwright 固化脚本隔离浏览器不稳定因素
  • 审核规则外置为 JSON/Markdown 配置，业务调整不改代码
  • 全程 SSE 流式输出，运营可实时观察审核过程

设计原则

  • 程序做程序擅长的事：Excel 解析、数据对比、表格构建用确定性代码
  • AI 做 AI 擅长的事：异常识别、风险研判、自然语言输出用 LLM
  • 人做最终决策：系统输出初审结果，运营确认后归档
  • 渐进式自动化：先固化高频操作，再逐步扩大 AI 自主权

Part 5

逐项解题

针对五大挑战，逐一说明解决方案与技术实现。

解题 1

BEM 数据自动化抓取 — 从手动登录到一键获取

方案：通过 Playwright 固化脚本自动化登录 BEM（智泊云）后台，解决验证码（ddddocr OCR 识别）、二次跳转等复杂认证流程。 一次审核自动获取三类报表：临停收入（按支付渠道分月明细）、月票收入（含趋势分析）、票种信息（内外部分类、在用数量）。 脚本以子进程隔离运行，浏览器崩溃不影响 Web 服务。

效果：运营无需手动登录后台，审核流程中 LLM 自动调用 fetch_bem_data 工具触发抓取，数据直接参与对比分析。

解题 2

智能 Excel 解析 — 标签定位替代固定行号

方案：解析器采用标签关键词查找（_find_label_row）而非硬编码行号，兼容不同版本格式的 Excel 模板。 自动识别业务类型（停车券 / 车位置换 / 折扣采买），支持多 Sheet 结构解析。 解析结果输出标准化 JSON，供后续 LLM 审核和数据对比使用。

效果：无论业务提交哪种模板格式，系统都能自动解析出结构化数据，消除格式差异带来的分析障碍。

解题 3

数据完整性前置检查 — 提交即检，减少退回

方案：必填字段定义外置为 fields_required.json，审核流程第一阶段即为完整性检查。 LLM 对照必填字段清单逐项校验，输出已填字段（通过）和缺失字段（需补充），未通过完整性检查的项目不予进入数据对比环节。

效果：业务提交后秒级反馈缺失项，无需等运营人工复核才发现问题，大幅减少退回往返次数。

解题 4

企业风险量化评估 — 自动化企查查核查

方案：LLM 通过 lookup_company 工具调用 Playwright 脚本查询启信宝，自动获取合作方的社保缴纳人数、经营状态、注册资本、成立日期。 同时在审核规则中明确：合同期限需在承包期限内、车位置换合同期需 ≥ 36 个月等硬性约束。 审核结果输出中对社保人数异常、期限不匹配等风险进行标注。

效果：原本需要运营手动查询、靠经验判断的风险点，现在由系统自动获取数据并给出风险提示，降低人为遗漏概率。

解题 5

AI 驱动的标准化初审体系

方案：将审核经验沉淀为三类可维护的配置文件： 审核清单（audit_checklist_*.md，分业务类型的检查项）、 风险规则（risk_rules.json，5% 容差、36 月最低期限等量化标准）、 输出模板（output_template.md，统一的状态标记和格式）。 LLM 按 "完整性检查 → 数据对比 → 风险评估 → 输出报告" 的标准化流程执行审核，System Prompt 由 8 个配置文件动态拼接。

效果：审核标准不再依赖个人经验，新人也能在 AI 辅助下产出高质量初审结果；规则调整只需改配置文件，无需动代码。

Part 7

审核执行流程

LLM 通过 Function Calling 自主编排审核步骤，无硬编码状态机。

Step 1 — 上传 & 解析

运营拖拽上传 Excel → excel_parser.py 自动识别业务类型、解析为结构化 JSON → 前端展示数据预览表格

Step 2 — System Prompt 组装

根据业务类型动态拼接 8 个配置文件：角色定义 + 字段规则 + 审核清单 + 风险规则 + 必填字段 + 完整性检查 + 业务审核 Skill + 输出模板

Step 3 — LLM 自主审核

LLM 接收结构化数据，通过 Function Calling 决定何时调用 fetch_bem_data（获取 BEM 报表）和 lookup_company（查询企业信息），流式输出分析过程

Step 4 — 程序化数据对比

_build_comparison() 构建 Excel vs BEM 的逐月对比：临停收入（5% 容差）、月票收入（全零跳过）、趋势分析（近 3 月 vs 前 3 月 >10% 预警）

Step 5 — 输出初审报告

LLM 输出结构化 Markdown 报告，每项标注状态（通过/预警/风险/人工确认/跳过），前端实时渲染并展示数据对比可视化图表

Part 8

技术亮点

关键设计决策与创新点。

LLM 无状态机编排

不预设审核步骤顺序，LLM 通过 Function Calling 自主决定调用工具的时机和顺序，更灵活地应对各种业务场景。

子进程隔离

BEM 和企查查脚本以子进程运行，Playwright 浏览器崩溃不影响 Web 服务稳定性。

验证码自动识别

集成 ddddocr 做 BEM 登录验证码 OCR，支持最多 10 次重试，解决登录自动化最大障碍。

标签定位解析

Excel 解析用关键词查找而非固定行号，天然兼容模板格式变化，降低维护成本。

规则外置可配

风险规则、必填字段、输出模板均为独立配置文件，业务调整只需改配置，无需改代码重新部署。

SSE 流式交互

全程 Server-Sent Events 实时推送审核状态、LLM 输出、工具调用进度，运营可观察审核过程而非等待黑箱结果。

Part 10

未来演进方向

当前系统已建立自动化审核基座，后续可在此基础上持续扩展。

非艾科车场数据接入

当前非艾科车场依赖业务提供数据。后续可对接更多停车管理系统（如科拓、捷顺），扩大自动数据获取覆盖面。

历史审核数据积累

将初审结果和运营最终判定归档，积累标注数据，为后续 Fine-tune 或 RAG 增强提供基础。

周边车场智能评估

接入地图 API 和公开停车费数据，为新车场自动生成周边竞品分析，辅助运营做综合评估。

多租户 & 权限管理

增加用户体系，支持业务自提交 + 运营审核的协作模式，实现端到端的流程闭环。
