## 二、需求文档内容

### 2.1 需求背景和流程.md

```
需求背景：
场景：资源置换业务评估测算
现业务流程：业务填写测算表，运营检查数据完整性，信息不全则退回补充。针对艾科车场，需核对后台临停、月票收入是否与测算表一致；非艾科车场则依据业务提供信息复核；若含自有支付渠道渠道的还需剔除后核对流水是否符合消耗比例。针对新车场或非艾科车场，运营需自行搜集周边车场收费信息，结合位置、业态等综合评估。
面临的挑战与问题：
1、新车场周边数据需逐个登录后台获取，缺乏统一报表；
2、各方提供的系统数据格式不一，分析困难；
3、业务提交前未自检，依赖运营复核；
4、需评估承包期、产权方经营风险；
5、缺乏智能初评体系，导致沟通反复。

业务流程：
1、先检查车场的资料信息即各个字段是否完整填写，如果字段信息齐全，则进行第2步；如果信息不齐全，则要求补全资料
2、再逐步检查上述的字段是否有风险，各个字段的检查如下
  临停：车场信息表的临停每个月收入的数据跟登录bem系统获取的临停收入-实收金额-微信对比
  月票：车场信息表的月票每个月收入的数据跟登录bem系统获取的月票收入对比
  合同有效年限（月）：需要在承包期限内
  券消耗年限（月）：需晚于上面的合同有效年限（月）
  临停-特殊：如果有自有小程序/app/ETC，需要用【总计-微信小程序（开放平台）-ETC】/总计=可消耗收入占比，如果停车券月均消耗额度≤可消耗收入占比*车场月均临停收入，则pass
  其他：分期代扣、CRM专项服务费、业务费，先打印出来，运营评估
  毛利率：计算出毛利率情况（项目整体毛利/停车券总价值），结合实际采买折扣输出权限范围
  合作客户主体：进行企查查，输出参保人数
```

### 2.2 资源置换评估操作.docx

```
1. 停车券业务：
1.1 信息收集表核对
- 合作客户主体：承包方时，需关注承包到期时间是否超出合同时间
- 是否存在自有小程序/ETC支付渠道：若是，则需要看剔除该渠道收入后的可消耗收入数据
- 依据合作客户主体，联动企查查等或者是ai帮我看其参保人数是多少人，输出人数给我

1.2 车场数据的复核：
1.2.1 AKE车场：则登录智泊云、智汇云后台，查看近一年的临停收入、月票收入、月票类型
1.2.2 非AKE车场——如果以下两个都没有，需要输出给业务进行获取。
  1.2.2.1 无提供车场历史的流水数据的情况下：在广告平台上搜索该车场周边500米内的车场（由近到远）
  1.2.2.2 提供了车场历史的流水数据：可能是图片、也可能是excel数据，需自行核对数据

1.3 测算表信息核对：
- 有分期代扣的话，需要输出业务了解分期代扣的额度/月多少
- CRM专项服务费与业务费不会同时共存，若有设备、服务置换金额的填写，则保留CRM专项服务费的值
- 合同有效年限（月）：需要在1.1的承包期限内
- 券消耗年限（月）：需晚于合同有效年限（月）
- 车场月租收入、临停收入需要跟1.1的临停月均、月票临停相等（若无涉及置换月票无需填写）
- 停车券月均消耗额度是否≤（【总计-微信小程序（开放平台）-ETC】/总计）—可消耗收入占比*车场月均临停收入数据
- 计算出毛利率情况（项目整体毛利/停车券总价值），结合实际采买折扣输出权限范围
- 权限范围为：超出总监权限时，请帮我调节测算表，输出可合作方案。——该评估模式会变，待确定

2. 车位置换业务：
2.1 信息收集表核对
- 合作客户主体：承包方时，需关注承包到期时间是否超出合同时间
- 是否存在自有小程序/ETC支付渠道：若是，则需要看剔除该渠道收入后的可消耗收入数据
- 是否允许张贴物料：若不允许，输出业务二次明确沟通

2.2 车场数据的复核：同停车券业务1.2

2.3 测算信息核对
- 信息完整度，该填都得填
- 设备服务置换金额若跟CRM结算价一致需要与业务确定下是否填错
- 对外办理的月卡费用：艾科车场的月票类型中需要为对外办理的月票的套餐（非VIP商户住户等）
- 合同有效年限（月）：需要在1.1的承包期限内；且需要3年及以上
- 计算出毛利率，输出对应的业务审核权限内
```

## 三、需求分析与关键点整理

### 3.1 业务全貌

核心场景是**资源置换评估测算**，分两条业务线：

| 业务线 | 停车券业务 | 车位置换业务 |
|--------|-----------|-------------|
| 主体 | 1.x | 2.x |
| 检查环节 | 资料收集 → 车场数据复核 → 测算表核对 | 资料收集 → 车场数据复核 → 测算信息核对 |

### 3.2 关键检查项（按流程顺序）

**第1步：资料完整性检查**
- 所有必填字段是否填写齐全
- 不齐则退回要求补全

**第2步：逐项风险检查**

| # | 检查项 | 数据来源 | 判定逻辑 |
|---|--------|---------|---------|
| 1 | 临停收入 | 上传资料 vs BEM系统 | 两者月均数据对比，一致则pass，不一致输出差异 |
| 2 | 月票收入 | 上传资料 vs BEM系统 | 同上 |
| 3 | 合同有效年限 | 上传资料 | 需在承包期限内（车位置换还要求≥3年） |
| 4 | 券消耗年限 | 上传资料 | 需晚于合同有效年限 |
| 5 | 自有支付渠道可消耗占比 | BEM系统 | [总计-微信小程序-ETC]/总计 → 算可消耗收入占比 |
| 6 | 停车券月均消耗额度 | 上传资料 | ≤ 可消耗占比×月均临停收入 则pass |
| 7 | 分期代扣/CRM专项服务费/业务费 | 上传资料 | 有则标出，由运营人工评估 |
| 8 | 毛利率 | 计算 | 项目整体毛利/停车券总价值 |
| 9 | 合作客户主体 | 企查查 | 输出参保人数 |
| 10 | 对外月卡费用（车位置换） | BEM系统 | 取非内部/VIP的月票套餐价格对比 |
| 11 | 设备服务置换金额（车位置换） | 上传资料 | 与CRM结算价一致时需确认是否填错 |

### 3.3 关键数据流向

```
运营上传资料（Excel/图片）
        ↓
   AI解析提取结构化数据
        ↓
BEM Skills（固化脚本）→ 获取临停/月票/月票类型等
        ↓
   AI对比分析 → 输出初审结果（按检查项逐条）
```

### 4.2 Harness（应用壳）职责

1. **AI Chat 对话界面** — 运营人员的唯一入口，支持上传文件、对话交互
2. **文件解析** — 接收上传的 Excel/图片，提取结构化数据
3. **Skill 调度** — 根据当前检查环节自动路由到对应 Skill
4. **上下文管理** — 维护整个审核会话的状态（上传的数据、BEM获取的数据、检查进度）

### 4.3 Skills（能力模块）设计

每个 Skill = `skill.md`（描述+触发规则+输出格式）+ `scripts/`（固化脚本）

| Skill | 触发时机 | 核心能力 | 固化脚本 |
|-------|---------|---------|---------|
| **bem-fetch** | 进入数据复核环节 | 登录BEM系统获取指定车场的临停收入、月票收入、月票类型数据 | `fetch_temp_parking.py` `fetch_monthly_ticket.py` `fetch_ticket_types.py` |
| **completeness-check** | 资料上传后 | 检查必填字段是否完整 | 无（纯AI逻辑） |
| **audit-parking-voucher** | 数据对比阶段（停车券业务） | 逐项执行停车券检查清单，输出初审结果 | 无（AI + bem-fetch数据） |
| **audit-spot-exchange** | 数据对比阶段（车位置换业务） | 逐项执行车位置换检查清单，输出初审结果 | 无（AI + bem-fetch数据） |
| **company-lookup** | 需要查询合作客户时 | 查企查查获取参保人数 | `query_qichacha.py` |

## 六、实现路径（运营视角的操作流程）

```
运营人员操作                    系统内部处理
─────────────                ────────────
1. AI Chat对话框
   上传车场资料
   (Excel/图片)         →   Harness 解析文件，提取结构化数据
                             ↓
                        触发 completeness-check skill
                        → 检查字段完整性
                        → 不全：输出缺什么，要求补充
                        → 全：继续
                             ↓
2. 确认业务类型          →   Harness 路由到对应 audit skill
   (停车券/车位置换)         ↓
                        触发 bem-fetch skill
                        → 运行固化脚本获取BEM临停/月票数据
                             ↓
                        触发对应 audit skill
                        → 逐项对比 + 风险检查
                        → 需要企查查时触发 company-lookup
                             ↓
3. 收到初审结果          ←   按检查项逐条输出：
                             ✅ 通过项
                             ⚠️  需关注项（附数据差异）
                             ❌ 风险项（附原因）
                             📋 人工确认项
```

## 七、关键设计决策

1. **skill.md 是核心** — 每个skill的md文件定义了AI的"行为说明书"，包含触发条件、执行步骤、输入输出格式。Harness靠读这些md来调度。

2. **BEM数据获取用固化脚本** — 脚本负责登录BEM、拉取数据、返回结构化JSON。AI不直接操作BEM系统，脚本做脏活，AI只分析。

3. **检查清单可配置化** — `risk_rules.json` 把检查规则外置，后续业务调整检查项时改配置不改代码。

4. **输出模板统一** — 审核结果按固定格式输出（通过/关注/风险/人工确认），运营一目了然。
